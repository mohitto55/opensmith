# /prd — 대화형 PRD 작성기

사용자의 대략적인 기획을 체계적으로 세분화하여 시스템 PRD 문서를 생성합니다.

## 동작 원리

```
사용자: /prd 캐릭터 대화 플랫폼
       ↓
Phase 1: 초기 분석 — 사용자 입력에서 파악 가능한 것 추출
       ↓
Phase 2: 세분화 질문 — 빠진 부분을 구조화된 질문으로 요청
       ↓
Phase 3: 반복 심화 — 답변 기반 추가 질문 (최대 3라운드)
       ↓
Phase 4: PRD 생성 — docs/prd/system-prd.md 작성
       ↓
Phase 5: 검토 및 확정 — 사용자 확인 후 최종 저장
```

---

## Phase 1: 초기 분석

사용자 입력 `$ARGUMENTS`에서 다음을 추출합니다:

```
파악된 것:
- 프로젝트명/기능명: ?
- 핵심 아이디어: ?
- 도메인: ?

파악 안 된 것 (Phase 2에서 질문):
- 타겟 사용자
- 핵심 기능 범위
- 비기능 요구사항
- 수익 모델
- 기술 제약
```

### 1-1. 기존 PRD 확인

```bash
ls docs/prd/ 2>/dev/null
```

- 기존 시스템 PRD가 있다 → 읽고 현재 요청과 비교
  - 갱신 필요 → Phase 2에서 변경 부분 집중 질문
  - 갱신 불필요 → "기존 PRD가 이미 이 내용을 포함합니다" 안내
- 없다 → 신규 작성 모드

---

## Phase 2: 세분화 질문

**한 번에 모든 질문을 던지지 않습니다.** 카테고리별로 묶어서 3~5개씩 질문합니다.

### 라운드 1: 핵심 정의

```
다음 질문에 답해주세요. 모르거나 나중에 정할 부분은 "미정"이라고 해도 됩니다.

🎯 핵심 정의
1. 이 프로젝트/기능이 해결하려는 핵심 문제는 무엇인가요?
2. 타겟 사용자는 누구인가요? (예: 10대 학생, 개발자, 소상공인 등)
3. 사용자가 이 서비스를 쓰는 주요 시나리오 3가지를 말해주세요.
4. 경쟁 서비스나 참고할 서비스가 있나요?
5. MVP에 반드시 포함되어야 할 기능 3가지는?
```

### 라운드 2: 범위 및 제약

답변을 분석한 후 다음 질문:

```
📐 범위 및 제약
1. 기술 스택 선호가 있나요? (프론트/백엔드/DB)
   없으면 제가 추천해드립니다.
2. 예상 사용자 규모는? (초기 100명? 1만명? 100만명?)
3. 수익 모델이 있나요? (광고, 구독, 프리미엄, 없음 등)
4. 출시 목표 시점이 있나요?
5. 이번에 명시적으로 하지 않을 것(Out of Scope)이 있나요?
```

### 라운드 3: 심화 (선택적)

이전 답변에서 모호한 부분이 있을 때만 실행:

```
🔍 심화 질문
- [이전 답변에서 모호했던 부분 1]에 대해 좀 더 구체적으로 설명해주세요.
- [기능 X]는 [시나리오 A]와 [시나리오 B] 중 어떤 방식으로 동작하나요?
- ...
```

### 질문 원칙

1. **이미 파악된 것은 다시 묻지 않는다**
2. **"미정"은 존중한다** — 강제로 결정을 요구하지 않음
3. **기술적 질문은 최소화** — PRD는 사용자 경험 중심
4. **선택지를 제공한다** — 열린 질문보다 "A vs B?" 가 더 빠름
5. **3라운드를 초과하지 않는다** — 질문 피로도 관리

## Phase 3: PRD 초안 생성

모든 답변을 종합하여 PRD를 작성합니다.

### 3-1. 저장 경로 결정

```
시스템 PRD (프로젝트 전체):
  docs/prd/system-prd.md

기능 PRD (개별 기능):
  docs/prd/features/[기능명-kebab-case].md
```

- `$ARGUMENTS`가 프로젝트 전체를 설명하면 → `system-prd.md`
- `$ARGUMENTS`가 특정 기능이면 → `features/[기능명].md`

### 3-2. PRD 작성 형식

```markdown
# PRD: [프로젝트/기능명]

> 작성일: YYYY-MM-DD
> 상태: Draft / Review / Approved
> 작성자: User + Claude

---
```

## Phase 4: 사용자 검토

PRD 초안을 사용자에게 보여줍니다:

```
PRD를 작성했습니다. 검토해주세요:

📋 docs/prd/system-prd.md

수정할 부분이 있으면 알려주세요.
확인되면 상태를 "Approved"로 변경합니다.
```

### 수정 요청 시
- 해당 섹션만 수정
- 다시 사용자에게 보여줌
- 최대 3회 수정 후에도 확정 안 되면: "나머지는 개발 중 구체화하겠습니다" 로 마무리

### 확인 시
- 상태를 `Approved`로 변경
- 기능 목록(Section 9)에 나열된 각 기능에 대해: "세분화 PRD가 필요하면 `/execute [기능명]` 실행 시 자동 생성됩니다" 안내
