# System Design Validator Skill

이 스킬은 대규모 시스템 설계 4단계 프로세스를 기반으로 시스템 설계의 완성도를 검증합니다.

---

## 검증 프로세스

### 1단계: 문제 이해 및 설계 범위 확정 검증

#### 기능 요구사항 체크리스트
다음 항목들이 명확히 정의되어 있는지 확인:

- [ ] 핵심 기능 목록이 정의되어 있는가?
- [ ] 각 기능의 우선순위가 결정되어 있는가?
- [ ] MVP (Minimum Viable Product) 범위가 명확한가?
- [ ] 지원하지 않을 기능도 명시되어 있는가?

#### 비기능 요구사항 체크리스트
- [ ] 가용성 (Availability) 목표가 정의되어 있는가?
- [ ] 응답 지연 시간 (Latency) 요구사항이 있는가?
- [ ] 일관성 (Consistency) vs 가용성 트레이드오프가 고려되었는가?
- [ ] 확장성 (Scalability) 요구사항이 있는가?
- [ ] 보안 요구사항이 정의되어 있는가?

#### 개략적 규모 산정
다음 수치들이 계산되어 있는지 확인:

**QPS (초당 쿼리 수) 계산**
```
QPS = DAU × 1인당 평균 요청 수 / 86400
Peak QPS = QPS × 피크 배율 (보통 2~3배)
```

- [ ] DAU (Daily Active Users)가 정의되어 있는가?
- [ ] 1인당 평균 요청 수가 추정되어 있는가?
- [ ] 읽기/쓰기 비율이 계산되어 있는가?
- [ ] Peak QPS가 고려되어 있는가?

**저장소 요구량 계산**
```
저장소 요구량 = 하루 생성 데이터 양 × 보관 기간
```

- [ ] 하루 생성되는 데이터 양이 추정되어 있는가?
- [ ] 데이터 보관 기간이 정의되어 있는가?
- [ ] 미디어 파일 (이미지, 비디오) 용량이 별도로 계산되어 있는가?
- [ ] 백업 및 복제 용량이 고려되어 있는가?

**대역폭 추정**
```
대역폭 = 평균 응답 크기 × QPS
```

- [ ] 평균 요청/응답 크기가 추정되어 있는가?
- [ ] 네트워크 대역폭이 충분한가?

---

### 2단계: 개략적 설계안 제시 검증

#### API 설계
- [ ] RESTful API 또는 GraphQL 등 API 스타일이 결정되어 있는가?
- [ ] 주요 API 엔드포인트가 정의되어 있는가?
- [ ] 요청/응답 포맷이 정의되어 있는가?
- [ ] API 버저닝 전략이 있는가?
- [ ] Rate Limiting이 고려되어 있는가?

#### 읽기 vs 쓰기 최적화
- [ ] 읽기/쓰기 비율에 따른 최적화 전략이 있는가?
- [ ] CQRS (Command Query Responsibility Segregation) 패턴 필요성이 검토되었는가?

#### 시스템 다이어그램
다이어그램에 다음 컴포넌트들이 포함되어 있는지 확인:

**필수 컴포넌트**
- [ ] 클라이언트 (Web, Mobile, Desktop)
- [ ] DNS
- [ ] CDN (정적 콘텐츠 서빙)
- [ ] 로드 밸런서 (Load Balancer)
- [ ] API Gateway
- [ ] 웹 서버 / 애플리케이션 서버
- [ ] 데이터베이스 (Primary/Replica)
- [ ] 캐시 (Redis, Memcached)

**선택적 컴포넌트 (요구사항에 따라)**
- [ ] 메시지 큐 (Kafka, RabbitMQ)
- [ ] 검색 엔진 (Elasticsearch)
- [ ] 객체 저장소 (S3, GCS)
- [ ] 모니터링 시스템
- [ ] 로깅 시스템

#### 데이터베이스 설계
- [ ] SQL vs NoSQL 선택이 합리적인가?
- [ ] 데이터 모델 (스키마)이 정의되어 있는가?
- [ ] 인덱스 전략이 수립되어 있는가?
- [ ] 파티셔닝/샤딩 전략이 있는가?
- [ ] 복제 전략이 정의되어 있는가?

#### 분산 ID 생성기
- [ ] 분산 ID 생성 방식이 정의되어 있는가?
- [ ] ID 생성 방식의 장단점이 고려되었는가?

---

### 3단계: 상세 설계 (추가 검증 항목)

#### 캐시 전략
- [ ] 캐시할 데이터가 식별되어 있는가?
- [ ] 캐시 정책이 정의되어 있는가?
- [ ] 캐시 만료 (TTL) 정책이 있는가?
- [ ] 캐시 일관성 전략이 있는가?
- [ ] Cache Stampede 대응 방안이 있는가?

#### 데이터 일관성
- [ ] 분산 트랜잭션 처리 방안이 있는가?
- [ ] Eventual Consistency가 허용되는 범위가 정의되어 있는가?
- [ ] 데이터 정합성 검증 방법이 있는가?

#### 메시지 큐
- [ ] 비동기 처리가 필요한 작업이 식별되어 있는가?
- [ ] At-least-once vs Exactly-once 배송 보장 레벨이 결정되어 있는가?
- [ ] Dead Letter Queue 처리 방안이 있는가?
- [ ] 메시지 순서 보장이 필요한가?

---

### 4단계: 마무리 (오류 처리 및 복원력)

#### 장애 시나리오
다음 장애 상황에 대한 대응 방안이 있는지 확인:

- [ ] 서버 장애 (Server Failure)
- [ ] 네트워크 장애 (Network Partition)
- [ ] 데이터베이스 장애
- [ ] 캐시 장애
- [ ] 메시지 큐 장애

#### 모니터링 및 알림
- [ ] 핵심 메트릭이 정의되어 있는가?
- [ ] 알림 임계값이 설정되어 있는가?
- [ ] 로깅 전략이 있는가?
- [ ] 분산 추적 (Distributed Tracing)이 고려되어 있는가?

#### 보안
- [ ] 인증/인가 메커니즘이 있는가?
- [ ] HTTPS가 적용되어 있는가?
- [ ] SQL Injection 방지가 되어 있는가?
- [ ] XSS, CSRF 방어가 되어 있는가?
- [ ] Rate Limiting / DDoS 방어가 있는가?
- [ ] 민감 데이터 암호화가 되어 있는가?

#### 성능 최적화
- [ ] 데이터베이스 쿼리 최적화가 고려되었는가?
- [ ] N+1 쿼리 문제가 해결되었는가?
- [ ] Connection Pooling이 사용되고 있는가?
- [ ] 비동기 처리가 적절히 사용되고 있는가?
- [ ] CDN 활용이 최대화되어 있는가?

---

## 검증 실행 방법

이 스킬을 실행하면 다음 작업을 수행합니다:

1. 프로젝트의 설계 문서를 읽습니다
2. 위의 4단계 체크리스트를 기준으로 검증합니다
3. 부족한 부분을 리포트로 작성합니다
4. 개선 제안을 제공합니다

---

## 출력 형식

검증 결과는 다음 형식으로 출력됩니다:

```
# 시스템 설계 검증 리포트

## 1단계: 문제 이해 및 설계 범위 확정
완료된 항목
- 핵심 기능 정의됨

미흡한 항목
- 비기능 요구사항 중 응답 지연 시간 목표 미정의
- QPS 계산 누락

누락된 항목
- Peak QPS 고려 안 됨
- 대역폭 추정 누락

## 2단계: 개략적 설계안
...

## 개선 제안
1. DAU와 평균 요청 수를 기반으로 QPS를 계산하세요
2. 읽기/쓰기 비율을 고려하여 캐시 전략을 수립하세요
...
```
