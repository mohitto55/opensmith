How It Works

전체 시스템 플로우

사용자가 /dev-cycle 하나를 입력하면, 아래 10단계가 자동으로 실행됩니다.
사람이 개입하는 건 1번(방향 지시)과 4번(설계 승인)뿐입니다.

1
👤 사용자가 방향을 지시한다
사용자는 코드를 작성하지 않습니다. 목표만 전달합니다.
예: "/dev-cycle 채팅에 이미지 전송 기능 추가해줘"
사용자 메시지는 컨텍스트 스택의 가장 위(Layer 5)에 도착합니다.
하지만 이미 Layer 1~4의 프로젝트 규칙, scaffold 패턴, 과거 팩트가 로딩된 상태입니다.
→ {{AI_MODEL}}는 짧은 지시만으로도 전체 맥락을 파악하고 작업을 시작합니다.
2
🧠 {{AI_MODEL}}가 컨텍스트를 조립한다
세션이 시작되면 SessionStart 훅 4개가 자동으로 실행되어 컨텍스트를 조립합니다.
자동 주입되는 것들:
① CLAUDE.md — 프로젝트 규칙, 빌드 커맨드, 기술 스택 (항상 로딩)
② Skills/Scaffold — frontend-patterns.md, backend-patterns.md, NEVER DO 규칙
③ inject-context.sh — Memory Bank에서 top-5 팩트 + pending 개선사항 주입
④ session-continuity.sh — 이전 세션의 WIP 커밋 메시지로 작업 이어가기
⑤ worktree-status.sh — 현재 활성 워크트리 목록

→ 사용자 메시지가 도착하기 전에 이미 풍부한 컨텍스트가 준비됨
3
📋 PRD를 확인/생성한다
/dev-cycle이 실행되면, 가장 먼저 docs/prd/에서 기존 PRD를 검색합니다.
없으면 자동으로 PRD를 생성하고, 있으면 업데이트 필요 여부를 확인합니다.
PRD 자동 생성 프로세스:
① docs/prd/ 디렉토리에서 기존 PRD 검색
② 없으면 → 사용자 요구사항 분석 → PRD 문서 자동 작성
③ 있으면 → 기존 PRD와 새 요구사항 diff → 업데이트 필요 시 수정

PRD에는 문제 정의, 기능 요구사항(FR), 비기능 요구사항(NFR), 성공 지표가 포함됩니다.
→ 이 PRD가 이후 모든 단계의 진실의 원천(Single Source of Truth)이 됩니다.
4
🏗️ 기술 설계 → 사용자 승인
PRD를 기반으로 기술 설계 문서를 작성하고, system-design-validator로 검증한 후
사용자의 승인을 받습니다. 이것이 사용자가 개입하는 유일한 두 번째 지점입니다.
기술 설계 프로세스:
① docs/design/ 에서 기존 설계 확인
② PRD의 FR/NFR → API 설계, 데이터 모델, 컴포넌트 구조 도출
③ system-design-validator 4단계 검증 (API 일관성, 보안, 확장성, 성능)
④ 사용자에게 설계 요약 제시 → 승인 대기

승인 후에만 구현 단계(Step 5)로 진행합니다.
→ 잘못된 방향으로 코드를 작성하는 것을 사전에 방지합니다.
5
📝 /agent-teams가 코드를 작성한다
승인된 설계를 기반으로 6개 역할 × 6 Phase 팀 오케스트레이터가 코드를 작성합니다.
scaffold 규칙 + NEVER DO + Memory Bank 팩트를 참고하여 처음부터 올바른 패턴으로 작성.
/agent-teams 오케스트레이터 발동:
6개 역할(설계자, DB 관리자, 프론트, 백엔드, QA, 팀 리더) × 6 Phase가 순차/병렬 실행

Phase 0 분석 → Phase 1 설계 → Phase 2 DB/타입 → Phase 3 구현(병렬) → Phase 4 QA → Phase 5 배포

프론트엔드 + 백엔드는 Phase 3에서 동시에 실행(Promise.all)되어 시간을 절약합니다.
6
⚡ 훅이 자동으로 검증한다
코드가 작성될 때마다(Write/Edit) PreToolUse + PostToolUse 훅이 자동 실행됩니다.
사람이 리뷰하기 전에 기계가 먼저 검증합니다.
PreToolUse (작성 전 차단):
• NEVER DO 위반 → 즉시 차단
• 보안 위반 (하드코딩 시크릿, wildcard CORS) → 즉시 차단
• 금지 라이브러리 import → 즉시 차단
• 파일 800줄 초과 → 즉시 차단

PostToolUse (작성 후 가이드):
• 컴포넌트 패턴 위반 → 수정 가이드 주입
• API 일관성 체크 → 경고
• 타입 에러 감지 → 자동 수정 에이전트 제안
• 성능 안티패턴 → 힌트 주입
7
🔧 에러가 나면 자동으로 고친다
검증에서 에러가 발견되면 3단계 에스컬레이션이 자동으로 처리합니다.
대부분 1~2단계에서 해결되며, 사람 개입은 극히 드뭅니다.
1단계 — 소프트 알림 (실패 1~2회)
에러 메시지를 컨텍스트에 주입 → {{AI_MODEL}}가 보고 직접 수정

2단계 — 에이전트 수정 (실패 3~5회)
독립 Opus 에이전트가 파일을 읽고, 에러를 분석하고, 자동 수정
메인 작업과 병렬 실행이라 작업 흐름이 끊기지 않음

3단계 — 강제 차단 (실패 6회+)
도구 실행 차단 + 텔레그램 알림 → 접근 방식 변경 요구
여기서만 사람이 개입합니다
8
🚀 빌드 검증 → 배포
코드가 검증을 통과하면 빌드 → OCI 업로드 → {{ORCHESTRATOR}} Pod 재시작 → 헬스체크까지 자동 실행.
배포 실패 시 deploy-fix-agent가 자동으로 원인 분석 및 수정을 시도합니다.
배포 파이프라인:
① {{BACKEND_FRAMEWORK}} build + npm run build — 프론트/백엔드 빌드 검증
② tar czf → 아카이브 생성
③ oci os object put → OCI Object Storage 업로드
④ {{ORCHESTRATOR}} rollout restart → {{ORCHESTRATOR}} Pod 재시작
⑤ 헬스체크 → 서비스 정상 확인

배포 실패 시 → deploy-fix-agent 자동 수정 → 재배포
9
🌐 /agent-browser가 실제 브라우저에서 QA 테스트
배포된 프로덕션 서버에 실제 브라우저(Headed mode)로 접속하여
자동으로 기능을 테스트합니다. 사람이 직접 클릭해서 확인하지 않습니다.
/agent-browser QA 프로세스:
① http://158.179.167.98 프로덕션 서버 접속
② Headed 모드 — persistent 프로필로 로그인 상태 유지
③ PRD의 기능 요구사항(FR) 기반으로 테스트 시나리오 자동 생성
④ 스크린샷 캡처 + 결과 리포트 생성

Stop 훅 (세션 종료 시 자동):
• fact-extraction.sh — 세션에서 의사결정/에러패턴 팩트 추출
• metrics-log.sh — 빌드 횟수, 에러 수 등 세션 메트릭 기록
• wip-commit.sh — 미커밋 변경이 있으면 WIP 커밋 자동 생성
10
💎 학습하고 강화한다 — 그리고 다시 1번으로
추출된 팩트가 Memory Bank에 저장됩니다. Fix 커밋이 감지되면
scaffold 파일에 새 NEVER DO 규칙이 자동 추가되어 같은 실수를 영원히 방지합니다.
자가개선 루프 (6단계 순환):
① 코드 작성 — Scaffold + Memory Bank 참고
② 검증 — 30개 훅이 자동 검사
③ 자동 수정 — 3단계 에스컬레이션으로 에러 해결
④ 배포 — Git → OCI → {{ORCHESTRATOR}} → 헬스체크
⑤ 학습 — Fix 커밋 감지 → "무엇이 문제였고 어떻게 해결했는지" 팩트 추출
⑥ 강화 — never-do.md AUTO-GENERATED 영역에 새 규칙 추가

→ 다음 세션부터 Layer 2(Scaffold)에 자동 반영
→ 다음 코드 작성 시 {{AI_MODEL}}가 강화된 규칙을 참조

∞ 무한 루프 — 이 사이클이 매 세션마다 반복되며 시스템이 점점 똑똑해집니다.

워크트리 자동화

각 기능 개발은 독립된 워크트리에서 진행됩니다.
메인 브랜치 오염 없이 병렬 작업이 가능하고, WIP 커밋으로 세션 간 연속성을 보장합니다.

워크트리 생성
claude -w → 자동으로 워크트리 생성
독립 물리 폴더 → 메인 브랜치와 완전 분리
여러 기능을 동시에 병렬 개발 가능
WIP 자동 커밋
세션 종료 시 Stop 훅 → wip-commit.sh
변경 감지 → 요약 + 의사결정 히스토리 포함
다음 세션에서 session-continuity.sh가 복원
/merge_worktree — 스쿼시 머지
① 미커밋 변경 확인 → WIP 커밋 생성 (필요 시)
② git checkout main → feature 브랜치에서 스쿼시 머지
③ WIP 의사결정 히스토리로 충돌 자동 해결
④ 모든 WIP 커밋을 합쳐 깨끗한 커밋 메시지 생성
⑤ 워크트리 제거 + 브랜치 삭제 (선택)
