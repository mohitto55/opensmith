# 5-Layer Architecture

## 개요

LoopyEra 아키텍처는 5개의 독립 레이어로 구성됩니다.
각 레이어는 명확한 책임을 가지며, 위 레이어일수록 사람에 가깝고, 아래 레이어일수록 자동화에 가깝습니다.

---

## Layer 1: 사용자 (Human)

### 역할
방향 설정, 피드백, 양질의 정보 제공. **HARD 차단 시에만** 텔레그램으로 알림을 받습니다.

### 속성
| 항목 | 값 |
|------|-----|
| 상호작용 빈도 | 최소화 (방향 설정 시에만) |
| 개입 트리거 | 3단계 에스컬레이션 도달 시 |
| 알림 채널 | 텔레그램 봇 |

### 하는 일
- 방향과 목표 설정 ("캐릭터 좋아요 기능 추가해줘")
- 양질의 피드백 제공 ("이 방식 대신 X 방식으로")
- 정보 제공 (아티클, 레포 URL)
- 텔레그램 알림 대응 (3단계 에스컬레이션)
- self-improve 제안 검토

### 안 하는 일
- 코드 작성
- 스킬 생성 (스킬이 스킬을 만듦)
- 버그 수정 (3단계 에스컬레이션이 처리)
- 테스트 실행 (훅이 자동 검증)
- 도구 실행 승인 (bypassPermissions)

## Layer 2: {{AI_MODEL}} (AI Brain)

### 역할
Opus 메인 스레드가 오케스트레이터로서 65개 스킬과 7개 전문 에이전트를 조율합니다.

### 구성

```
{{AI_MODEL}} Opus 4.6 (Main Thread)
├── 팀 오케스트레이터 (/team)
│   ├── 설계자 에이전트
│   ├── DB 관리자 에이전트
│   ├── 프론트엔드 에이전트
│   ├── 백엔드 에이전트
│   ├── QA 에이전트
│   └── 팀 리더 (메인 스레드)
├── 65개 스킬
│   ├── 개발 스킬 (agent-teams, dev-cycle, deploy, ...)
│   ├── 검증 스킬 (system-design-validator, qa-test, ...)
│   ├── 도구 스킬 (color-picker, agent-browser, ...)
│   └── 자가개선 스킬 (/self-improve, /scaffold-update, ...)
└── bypassPermissions 모드
    └── 승인 없이 도구 실행 가능
```

### 속성
| 항목 | 값 |
|------|-----|
| 모델 | {{AI_MODEL}} Opus 4.6 (1M context) |
| 권한 | bypassPermissions — 승인 없음 |
| 오케스트레이션 | 팀 리더로서 6역할 조율 |
| 전문가 수 | 7개 (설계, DB, 프론트, 백엔드, QA, 브라우저, 보안) |

## Layer 3: 훅 (Guard Rails)

### 역할
11개 이벤트에 30개 훅이 매핑되어 코드 품질을 자동으로 보장합니다.

### 훅 유형별 동작

```
[코드 작성]
    │
    ▼
┌─ HARD Gate ─┐     ┌─ SOFT Hook ──────┐
│ 타입 체크    │     │ 컨텍스트 주입     │
│ 린트 체크    │     │ 연속성 노트       │
│ 보안 체크    │     │ 에러 패턴 경고    │
│ scaffold 체크│     │ 가이드 메시지     │
└──────────────┘     └──────────────────┘
    │ 실패 시              │ 항상 실행
    ▼                      ▼
┌─ Agent Hook ─┐     [메인 스레드 계속]
│ Opus 에이전트 │
│ 에러 분석     │
│ 자동 수정     │
│ 메인과 병렬   │
└──────────────┘
```

### 속성
| 항목 | 값 |
|------|-----|
| 총 훅 수 | 30개 |
| 이벤트 유형 | 11개 |
| HARD 게이트 | 8개 (차단형) |
| Agent 훅 | 6개 (자동 수정형) |
| SOFT 훅 | 16개 (가이드형) |

→ 상세: [Hook System](./hook-system.md)

## Layer 4: 메모리 (Memory Bank)

### 역할
시맨틱 검색 기반 2,600+ 팩트 저장소. 과거 의사결정, 에러 패턴, 프로젝트 제약사항을 자동으로 주입합니다.

### 데이터 흐름

```
[대화 발생]
    │
    ├──→ 대화 인덱싱 파이프라인
    │    JSONL → 파서 → Embeddings → SQLite+vec → 시맨틱 검색
    │
    └──→ 팩트 추출 파이프라인
         Session End → Haiku LLM → 통합 → Facts DB → Ontology
```

### 속성
| 항목 | 값 |
|------|-----|
| 임베딩 모델 | all-MiniLM-L6-v2 (384-dim) |
| 벡터 DB | SQLite + sqlite-vec |
| 팩트 목표 | 2,600+ |
| 대화 목표 | 50K+ |
| 검색 방식 | cosine similarity + FTS hybrid |

→ 상세: [Memory Bank](./memory-bank.md)

## Layer 5: 인프라 (Infrastructure)

### 역할
Git, 빌드 도구, 데이터베이스, 외부 통신 등 기반 시스템을 제공합니다.

### 구성 요소

| 도구 | 용도 |
|------|------|
| **Git** | 버전 관리, 워크트리, WIP 커밋 |
| **SQLite** | Memory Bank 저장소, 팩트 DB |
| **npm/tsc** | 프론트엔드 빌드 + 타입 체크 |
| **{{BACKEND_FRAMEWORK}}** | 백엔드 빌드 |
| **vitest** | 프론트엔드 테스트 |
| **Telegram** | 3단계 에스컬레이션 알림 |
| **MCP** | Model Context Protocol 서버 |

### 인프라 연결도

```
                    ┌──────────────────┐
                    │  Telegram Bot     │
                    │  (에스컬레이션)    │
                    └────────▲─────────┘
                             │ 3단계 알림
┌──────────┐    ┌────────────┴───────────┐
│   Git    │◄──►│    Claude Code (Local)  │
│ Worktree │    │    ┌─────────────────┐  │
│          │    │    │ SQLite + vec    │  │
└──────────┘    │    │ (Memory Bank)   │  │
                │    └─────────────────┘  │
                │    ┌─────────────────┐  │
                │    │ npm/tsc/{{BACKEND_FRAMEWORK}}  │  │
                │    │ (빌드 도구)      │  │
                │    └─────────────────┘  │
                └─────────────────────────┘
                             │
                             ▼
                ┌─────────────────────────┐
                │  OCI {{ORCHESTRATOR}} Cluster        │
                │  (프로덕션 배포)         │
                └─────────────────────────┘
```

## 레이어 간 상호작용

### 상향 (Bottom-Up) 흐름: 에스컬레이션
```
인프라(L5) → 에러 발생
  → 훅(L3) → 자동 수정 시도
    → 실패 시 {{AI_MODEL}}(L2) → 에이전트 수정 시도
      → 실패 시 사용자(L1) → 텔레그램 알림
```

### 하향 (Top-Down) 흐름: 지시
```
사용자(L1) → "기능 X 추가해줘"
  → {{AI_MODEL}}(L2) → /team 오케스트레이터 실행
    → 훅(L3) → 코드 품질 자동 검증
      → 메모리(L4) → 과거 패턴 참조
        → 인프라(L5) → Git 커밋 + {{ORCHESTRATOR}} 배포
```

### 순환 (Loop) 흐름: 자가개선
```
코드 작성(L2) → 검증(L3) → 에러 → 수정(L2) → 배포(L5)
  → 학습(L4 팩트 추출) → 강화(L2 scaffold 업데이트)
  → 다음 코드 작성 시 같은 실수 방지
```
